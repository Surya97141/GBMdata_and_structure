Persistent Segment Tree

A segment tree that never forgets — every update creates a new version while sharing unchanged nodes, enabling time-travel queries on any historical state.

A Persistent Segment Tree extends a regular segment tree by preserving all historical versions after updates. Instead of modifying nodes in place, each update creates at most O(log n) new nodes along the changed path and reuses all unchanged nodes from the previous version. The result: a branching tree of versions where each root points to a complete, self-consistent snapshot.

Think of a Git repository. Each commit creates a new tree object, reusing unchanged blobs and trees. You can checkout any commit (version) and see the exact file state at that point. A persistent segment tree does the same for arrays: update one element, get a new version — but "old commits" remain fully queryable in O(log n) time without storing n elements each time.

The key insight: only the nodes on the root-to-leaf path change (O(log n) nodes). Every other subtree is identical in both versions — represented by the same node object in memory, pointed to by both version trees. This sharing makes each update O(log n) space instead of O(n).

Update
O(log n) time+space
Query (any version)
O(log n)
Total Space
O(n + q·log n)
Use Cases
Offline queries, Undo history, K-th order stats
