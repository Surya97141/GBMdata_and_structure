Centroid Decomposition

Recursively find each subtree's "balance point" — a centroid that guarantees at most O(log n) layers, enabling efficient path counting and distance queries.

Centroid Decomposition processes a tree by repeatedly finding and removing the centroid of the current component. A centroid of a component of size n is a node whose removal leaves no component larger than n/2. Such a node always exists and can be found in O(n). After removing the centroid, the process recurses on each resulting sub-component.

Think of finding the geographic center of a city so you can divide it into fair districts. The centroid is the "fairest" dividing point — after removing it, no piece is more than half the original. Recursing guarantees the tree decomposes into at most O(log n) layers because each sub-problem is at most half the size of the parent. This is the same reason binary search runs in O(log n) steps.

Centroid decomposition enables solving "count paths of length exactly k" or "closest node with property P on any path through this centroid" efficiently. All paths in the tree pass through exactly one centroid in the decomposition tree, so O(n log n) total work suffices for many path problems.
